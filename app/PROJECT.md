# 競馬トラックバイアス分析アプリ — プロジェクト全体まとめ

最終更新: 2026-05-29

## 1. 目的
書籍『競馬場コース事典2』のトラックバイアス（各競馬場・各コースの有利条件）と、
血統書籍(2024年版)の系統知識をベースに、毎週の中央競馬の出馬表（枠順確定後）から
**各レースの有利な馬を自動で抽出**する。
- フェーズ1: Claudeとの対話ベースで運用（現在ここ）
- フェーズ2: 恒常運用が固まったらアプリ化

## 3. データソースと取得方法
| ソース | 取得項目 | URL/方法 | キャッシュTTL |
|---|---|---|---|
| netkeiba race_list | 開催日の全race_id | `race.netkeiba.com/top/race_list_sub.html?kaisai_date=YYYYMMDD` | 1h |
| netkeiba shutuba | 出馬表・枠順・馬番・騎手 | `race.netkeiba.com/race/shutuba.html?race_id=...` | 1h |
| netkeiba 馬詳細 | 直近5走（場/距離/着順/上がり/通過順） | `db.netkeiba.com/horse/result/{id}/` | 24h |
| netkeiba 血統表 | 5代血統（父・母父・父系祖先） | `db.netkeiba.com/horse/ped/{id}/` | 24h |

- race_id 体系: `YYYY` + 場コード(05=東京,08=京都) + 開催回 + 日目 + R番号（12桁）
- 文字コードは EUC-JP（取得時に UTF-8 変換）

## 4. スコアリングロジック（v0.3）
`analyze.py evaluate_horse()` が各馬を加点:

| 項目 | 加点 | 判定 |
|---|---|---|
| 枠順バイアス一致 | +2 | コースの gate_bias（内枠/外枠/1枠有利）と馬番の内中外区分が一致 |
| 父が注目血統 | +3 | コースの sire_favored と一致（系統 or 個別馬名） |
| 母父が注目血統 | +2 | 同上（母父） |
| 前走条件一致 | +3 | 同距離/距離短縮延長/着順/上がり最速/先行差し/重賞名/前走場距離 |
| 軽斤量 | +1 | レース内最小斤量+0.5kg以内（恵まれ評価） |
| 上がり最速級 | +3 | 直近3走の最速上がりがレース内トップ級（能力代理） |
| 複勝安定 | +2 | 直近5走の複勝率60%以上 |
| 上位クラス経験 | +2 | 今走より上のクラスを経験（通用実績） |
| 展開脚質適性 | +2 | レース内の前走脚質集計から想定ペースを判定し合致脚質を加点 |
| 馬場状態適性 | +2 | 天候・馬場で有利脚質を判定（良ダ=前残り先行／道悪ダ=差し／道悪芝=前内）し合致馬を加点 |
| Calimero穴知見 | -2〜+10 | カリメロ@穴馬オタクの過去1年予想実績(複勝率/単回収)から導出した S3〜S10 シグナルの合算。`weights["cal"]`(現在1.0)で全体係数を調整(0で無効)。詳細→ `notes/calimero_integration_plan.md` |

#### Calimero穴知見の内訳
過去85記事1,978pickの結果照合(`notes/calimero_analysis.md`)から、複勝率/単回収率が有意に高かった条件を抽出。各馬で該当する分だけ加点(/減点)。

| Sig | 条件 | 加点 | 根拠 |
|---|---|---:|---|
| S3 | ダート | +1 | ◎ダ単回収396% vs 芝260% |
| S4 | 距離短縮(-400〜-100m) / 延長(+100〜+400m) | +2 | 短縮529% / 延長442% |
| S5 | 騎手減量印(☆▲△◇★) | +2 | 減量含む◎ 単回収458% |
| S6 | 出走頭数 ≤ 10 | +1 | 少頭数◎ 複勝26.9% |
| S8 | レース番号 6/9/12R | +1 | 6:376% 9:444% 12:397% |
| S8 | レース番号 11R | -1 | 11R◎ 単回収230%(穴薄) |
| S9 | 小倉/福島開催 | +1 | 小倉679% 福島487% |
| S9 | 札幌開催 | -1 | 札幌◎ 単回収134% |
| S10 | 1枠 or 大外枠 | +1 | 1枠20.9% 大外23.2%複勝 |

※ S1/S2(想定人気帯)は前日オッズ未確定により**不採用**。
※ 場別ブースト(S9)はサンプル小(◎n=42)のため半年毎に再較正前提。

### 馬場状態の扱い（重要）
- 出馬表に天候・馬場・クッション値が載っていれば自動取得（`fetch_shutuba`）。ただし**netkeibaは発走直前まで載せない**ため前日〜朝は通常None。
- `analyze_race(baba=)` の優先順位: **明示引数 > 出馬表の実データ > 良(デフォルト)**。
- 道悪前提のheadline（例「馬場が湿ると差し有利」）は、良馬場時に**自動で『不適用』注記**を付け、馬場に応じた脚質補正に置き換える。
- 当日の実馬場で再評価する場合: `recommend.py YYYYMMDD --baba 稍` のように上書き。

※オッズ・人気は当日変動で前日未確定のため取得スキップ（HTMLには欄あり）。斤量は前日取得可。
※能力バイアス4種は`build_race_context()`でレース内相対計算（追加スクレイピング不要、既存horses_*.json使用）。導入で「上位3頭が同点（横一線）」のレースが24→0に解消。

- 内中外区分: 馬番を頭数で3分割
- 前走条件は courses.json の rules テキストを `parse_rule()` で分解して判定

## 5. 血統マッチングの設計（重要）
### 問題（旧ロジック）
「キングカメハメハ系」を `系`削除して `キングカメハメハ` で部分マッチ →
ロードカナロア・ルーラーシップ等の系統メンバーを全て取りこぼしていた。
系統エントリは全sire_favoredの19%・12コースに存在し、実害大だった。

### 解決（v0.3）
- **書籍を権威源**として `lineage.json`（種牡馬→大系統/小系統/型）を構築
- ネット情報は"有象無象"を避け、**血統表の父系祖先という事実のみ**で大系統を自動判定 → `lineage_fallback.json`
- `sire_matches()`: favoredが`○○系`なら lineage の大系統/小系統と照合、個別馬名は完全一致
- カバレッジ: ユニーク種牡馬の約80% / 出走数加重 約88%
- **完全版**（個別プロフィール約200頭の全読・型/適性付与）はバックグラウンドタスクで進行中

## 7. 出力の読み方（recommend_*.md）
```
1. 11番 メイショウオウギ（牡3/角田河）**18%** [score 8]
   └ 外枠(11)有利; 父シニスターミニスター=注目血統; 前走後方差し(通過10-10)
```
- **%** = レース内バイアス適合スコアの占有率（**勝率予測ではない**）。
  トラックバイアス・血統・前走条件への合致度の相対指標。
- score = 加点合計（5節の配点）
- └ = 加点された具体的理由（最大3件表示）

## 8. 既知の限界 / バックログ
1. 完全版 lineage.json（個別プロフィール全読・型/適性）← バックグラウンド進行中
2. 障害コース（例 東京ダ3000m）は courses.json 未登録 → 対象外表示
3. クッション値・含水率は未取得（道悪ルールは前走馬場で近似）
4. 上がり最速は絶対値<34.0で近似（同レース内1位判定は未実装）
5. 的中ログ feedback.jsonl による重み調整（未着手）
6. %はスコア占有率であり統計的勝率ではない → 過去成績での較正が将来課題

## 9. トラブルシュート（発生した事象と修正）

### ★枠順が反映されない（最重要バグ・修正済み）
- 症状: 枠順確定後も全馬 umaban="" になり、枠順スコアが入らない。
- 原因: netkeibaは**枠順確定でセルのclass名が変わる**。
  確定前 `<td class="Waku Txt_C">`（空）→ 確定後 `<td class="Waku1 Txt_C"><span>1</span>`
  （枠/馬番がclass名に付く）。パーサが `.Waku`/`.Umaban`（完全一致）で取っていたため
  確定後の `Waku1`/`Umaban3` 等にマッチせず空になっていた。
- 修正: `td[class^="Waku"]` / `td[class^="Umaban"]`（前方一致）に変更。

### データ縮退（netkeibaの未来日付ボラティリティ）
- 症状: 再取得すると一部レースが3頭プレースホルダーに縮退、全レースが消えることも。
- 原因: 未来日付ではnetkeibaの出馬表/レース一覧が不安定。
- 修正: fetch_shutuba.py に**和集合マージ＋良い方を残す**安全機構を追加
  （頭数・枠番数が多い方を採用。既存レースは消さない）。

### 京都芝1400m等が「未登録コース」になる
- 原因: netkeibaが内/外を明示しない場合 variant=None になり、内/外しか存在しない
  コースで course_key が外れる。
- 修正: course_key に 内→外 のフォールバックを追加。

### 旧トラブル（参考）
- **zsh の glob no-match エラー**: `rm cache/shutuba_2026053*.html` のようにマッチ無しの
  glob を書くと zsh は文字列をそのまま渡さずエラー停止する。
  → キャッシュは `shutuba_<race_id>.html` 命名（日付ではない）。`rm -f cache/shutuba_*.html` を使う。
- **並列ツール呼び出しの巻き添えキャンセル**: 1つでも失敗すると同バッチ全部が Cancelled になる。
  依存・脆い処理（rm→fetch→analyze）は逐次実行する。
- **枠順確定前データの混入**: 枠順発表前は全馬 umaban="" のため理由が「外枠(?)」になる。
  枠順確定後に shutuba を取り直せば解消。
```
